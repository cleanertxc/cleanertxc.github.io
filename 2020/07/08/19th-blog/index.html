Chapter 20 Paging_Smaller Tables

Created2020-07-08Updated2020-07-27

linear page tables在进程多的时候占用的内存空间过大，因此我们要想办法减小他。

20.1 Simple Solution：Bigger Pages

纯粹的加大page的大小，这样page table entry的数量也会相应的减少，但是不足之处也很明显，page太大的话，会造成很多的internal fragmentation，因为应用实际上用不到这么大的空间。

20.2 Hybrid Approach：Paging and Segments

把一个process的page table弄成三个segments（简化版），分别是code、heap、stack。这样做还是有一些问题，第一，如果我们有一个很大的但是用的很少的heap，还是会造成大量的浪费。第二，page table的尺寸大小一直在变，所以还是会造成external fragmentation。

20.3 Multi-level Page Tables

引入一个新的概念叫page directory，可以告诉你这个page是不是valid，如果是valid，则告诉在物理内存哪个位置，否则直接不分配内存。同时这个page directory可以有多层，主要是为了让每个directory都可以完整的放在一个page里面，所以不能过大。

20.4 Inverted Page Tables

之前是一个process一个table，现在变成了所有的process共用一个page。
